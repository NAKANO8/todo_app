# Implementation Plan

- [ ] 1. Foundation: スキーマとデータ型の準備
- [ ] 1.1 Prismaスキーマに担当者列を追加する
  - `Todo`モデルに`assigneeId`（nullable、`users.id`へのFK）を追加し、`User`側に別名の逆参照リレーションを追加する
  - 新規マイグレーションファイルを作成し、既存Todoの`assignee_id`が全てNULLのまま適用されることを確認する
  - 観測可能な完了状態: マイグレーション適用後、`todos`テーブルに`assignee_id`列が存在し、既存行は全てNULLである
  - _Requirements: 1.5_

- [ ] 1.2 (P) バックエンドのTodo型定義を拡張する
  - `Todo`型に`assigneeId: number | null`を追加し、一覧・詳細表示用に担当者名を含む型（`TodoWithAssignee`）を新設する
  - 観測可能な完了状態: 型チェック（`pnpm build`）が新しいフィールドを含めて成功する
  - _Requirements: 1.5, 5.1, 5.2_
  - _Boundary: todo.ts types_

- [ ] 2. Core: 担当者の永続化とバリデーション
- [ ] 2.1 Todoリポジトリで担当者の読み書きを実装する
  - タスク作成時・更新時に担当者IDを保存・解除できるようにする（担当者省略時は既存値を変更しない部分更新の挙動を維持する）
  - 既存の作成者スコープ（本人以外のタスクを操作できない）は変更しない
  - 観測可能な完了状態: 担当者ID付きでタスクを作成・更新すると、DB上の該当レコードに担当者IDが反映される
  - _Requirements: 1.1, 1.2, 1.4, 1.5, 7.1, 7.2_
  - _Depends: 1.1_
  - _Boundary: TodoRepository_

- [ ] 2.2 担当者候補の解決とバリデーションをTodoサービスに実装する
  - タスク作成者が現在所属するグループのメンバー一覧（作成者自身を含む）を解決する処理を追加する
  - 担当者候補一覧を返す処理を追加する
  - 作成・更新時に指定された担当者が候補集合に含まれない場合はエラーとして拒否し、担当者を設定しない
  - 担当者未指定・解除（null指定）はエラーにせずそのまま許可する
  - 観測可能な完了状態: 候補外のユーザーIDを担当者に指定した作成・更新要求がエラーとなり、タスクの担当者が変更されない
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 2.1, 2.2, 2.3, 3.1, 3.2, 4.1, 6.1, 6.2_
  - _Depends: 2.1_
  - _Boundary: TodoService_

- [ ] 2.3 一覧・詳細表示への担当者情報の付与をTodoサービスに実装する
  - タスク一覧・単一タスク取得の結果に、担当者が設定されていればその表示名を、未設定であれば未設定である旨を含める
  - 観測可能な完了状態: 担当者設定済みタスクの取得結果に担当者名が含まれ、未設定タスクでは担当者名がnullとして返る
  - _Requirements: 5.1, 5.2_
  - _Depends: 2.2_
  - _Boundary: TodoService_

- [ ] 3. Core: APIエンドポイントの拡張
- [ ] 3.1 担当者候補一覧エンドポイントを追加する
  - 認証済みユーザーが自分の担当者候補一覧を取得できるエンドポイントを追加する（ロール制限は設けない）
  - 観測可能な完了状態: 認証済みリクエストで担当者候補一覧が返り、未認証リクエストは拒否される
  - _Requirements: 2.2_
  - _Depends: 2.2_
  - _Boundary: TodoController, todos.route.ts_

- [ ] 3.2 タスク作成・編集APIに担当者指定を統合する
  - タスク作成・更新のリクエストで担当者IDを受け取り、不正な型（数値・null以外）をスキーマ検証段階で拒否する
  - サービス層のバリデーション結果に応じたエラーレスポンス（不正な担当者、存在しないタスク）を返す
  - 観測可能な完了状態: 担当者ID付きのタスク作成・更新リクエストが成功し、不正な担当者指定・存在しないタスクへの操作がエラーレスポンスを返す
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 6.1, 6.2, 6.3_
  - _Depends: 2.2_
  - _Boundary: TodoController, todos.route.ts_

- [ ] 3.3 タスク一覧・詳細APIのレスポンスに担当者情報を含める
  - 一覧・詳細取得エンドポイントのレスポンスに担当者ID・担当者表示名を含める
  - 観測可能な完了状態: 一覧・詳細エンドポイントのレスポンスJSONに担当者情報フィールドが含まれる
  - _Requirements: 5.1, 5.2_

# Requirements Document

## Project Description (Input)
会社の管理者（`role`が`admin`のユーザー）は、現状「誰が自社のシステムにアクセスできるか」を自分でコントロールする画面を持っていない。ユーザーの一覧を見る手段も、退職者や不要になったアカウントを無効化する手段も存在せず、必要な場合は開発者がDBを直接操作するしかない。

このため、管理者画面を実装し、以下を可能にする。
- ユーザー一覧の表示（各ユーザーのロール・状態が分かる）
- ユーザーの権限（role: `admin`/`member`）変更
- ユーザーアカウントの無効化（退職者等、システムへのアクセスを止めたいユーザー向け）

前提として、`role`カラムの追加とAPI応答への反映（`admin-role` spec）、および管理者による強制セッション無効化API（`DELETE /admin/sessions/:userId`、`session-invalidation` spec）は既に実装済みであり、本specはこれらの基盤の上にUIおよび関連APIを構築するものである。アカウント無効化は、ログイン拒否だけでなく既存セッションの強制ログアウトも伴うべきか、実装時に検討する。

認可（グループ/チーム単位のアクセス制御）はスコープ外とし、別イテレーションで扱う。

## Introduction
現在、管理者はシステムに登録されている全ユーザーを一覧で確認する手段を持たず、各ユーザーの権限（ロール）を変更したり、退職者や不要になったアカウントを無効化したりする操作も行えない。必要な場合は開発者がデータベースを直接操作するしかなく、企業導入において必須視される最低限のユーザー管理ができない状態にある。本機能は、管理者がユーザー一覧を確認し、ロールを変更し、アカウントを無効化・再有効化できる操作手段を提供する。

## Boundary Context (Optional)
- **In scope**: 管理者によるユーザー一覧の参照（ロール・アカウント状態を含む）、ユーザーのロール変更、アカウントの無効化と再有効化、無効化に伴う既存セッションの強制ログアウト、これらの操作を管理者のみに限定すること、唯一の管理者による自己降格・自己無効化の防止
- **Out of scope**: グループ/チーム単位のアクセス制御（認可）の実装、ユーザー一覧の検索・絞り込み・ページネーション、アカウントのデータ削除、無効化操作の監査ログ記録、無効化された本人への通知、初期管理者アカウントの作成フロー
- **Adjacent expectations**: 本機能は、既に導入済みの`role`（管理者/一般ユーザー）区別を用いて実行権限を判定する。アカウント無効化時の強制ログアウトは、既に提供されている管理者向けセッション強制無効化の能力を利用することを期待するが、その具体的な連携方法は設計フェーズで定める。ユーザー一覧はグループや組織による絞り込みを行わず、システムに登録されている全ユーザーを対象とする。

## Requirements

### Requirement 1: 管理者によるユーザー一覧の参照
**Objective:** As a 管理者, I want 登録されている全ユーザーの一覧を、それぞれのロールとアカウント状態と共に確認したい, so that 誰がシステムにアクセスできるかを自分で把握できる

#### Acceptance Criteria
1. When 管理者がユーザー一覧を要求する, the ユーザー管理機能 shall 登録されている全ユーザーについて、各ユーザーのロールとアカウント状態（有効/無効）を含む一覧を提示する。
2. The ユーザー管理機能 shall ユーザー一覧に、システムに登録されている全ユーザーを、グループや組織による絞り込みを行わずに含める。

### Requirement 2: 管理者によるユーザーのロール変更
**Objective:** As a 管理者, I want 特定ユーザーのロール（管理者/一般ユーザー）を変更したい, so that 誰にどの権限を持たせるかを自分でコントロールできる

#### Acceptance Criteria
1. When 管理者が特定ユーザーのロールを変更する, the ユーザー管理機能 shall 対象ユーザーのロールを指定された値に更新する。
2. When ユーザーのロールが変更される, the ユーザー管理機能 shall 変更後、そのユーザーが次にアカウント情報を取得した際に新しいロール値を反映する。

### Requirement 3: 新規アカウントおよび既存アカウントのアカウント状態の既定値
**Objective:** As a システム運用者, I want 新規登録されたアカウントおよび本機能導入前から存在するアカウントが、既定で有効な状態であってほしい, so that 本機能の導入によって既存・新規ユーザーの利用が妨げられない

#### Acceptance Criteria
1. When 新規ユーザーが登録される, the ユーザー管理機能 shall そのアカウントの状態を「有効」とする。
2. When 本機能が導入される, the ユーザー管理機能 shall 導入前から存在するすべてのアカウントの状態を「有効」とする。

### Requirement 4: 管理者によるアカウントの無効化
**Objective:** As a 管理者, I want 特定ユーザーのアカウントを無効化したい, so that 退職者や不正利用者のアクセスを即座に止められる

#### Acceptance Criteria
1. When 管理者が特定ユーザーのアカウントを無効化する, the ユーザー管理機能 shall 対象アカウントの状態を「無効」に変更する。
2. When アカウントが無効化される, the ユーザー管理機能 shall そのユーザーに紐づく既存のログイン済みセッションを強制的に終了させる。
3. If 無効化されたアカウントでログインが試行される, then the ユーザー管理機能 shall そのログインを拒否する。

### Requirement 5: 無効化されたアカウントの再有効化
**Objective:** As a 管理者, I want 無効化したアカウントを後から再度有効化したい, so that 誤操作や再雇用などの状況に対応できる

#### Acceptance Criteria
1. When 管理者が無効化されたアカウントを再有効化する, the ユーザー管理機能 shall 対象アカウントの状態を「有効」に戻す。
2. When アカウントが再有効化される, the ユーザー管理機能 shall そのアカウントでの以降のログイン試行を、無効化されていない他のアカウントと同様に受け付ける。

### Requirement 6: 操作対象の実行権限の限定
**Objective:** As a システム運用者, I want ユーザー一覧の参照・ロール変更・アカウントの無効化・再有効化を管理者のみに限定したい, so that 一般ユーザーがこれらの操作を行えないようにする

#### Acceptance Criteria
1. If 管理者権限を持たないユーザーがユーザー一覧の参照、ロール変更、アカウントの無効化、または再有効化のいずれかを要求する, then the ユーザー管理機能 shall その要求を拒否する。
2. The ユーザー管理機能 shall ユーザー一覧の参照、ロール変更、アカウントの無効化、再有効化のいずれについても、管理者権限を持つユーザーからの要求のみを許可する。

### Requirement 7: 唯一の管理者に対する自己操作の保護
**Objective:** As a システム運用者, I want システムに管理者が1人しか存在しない場合、その管理者が自分自身を降格または無効化できないようにしたい, so that 管理者画面を誰も使えなくなる状態を防げる

#### Acceptance Criteria
1. If 管理者権限を持つユーザーが自分自身のロールを一般ユーザーに変更しようとしており、かつシステムに他に管理者権限を持つユーザーが存在しない, then the ユーザー管理機能 shall その変更を拒否する。
2. If 管理者権限を持つユーザーが自分自身のアカウントを無効化しようとしており、かつシステムに他に管理者権限を持つユーザーが存在しない, then the ユーザー管理機能 shall その無効化を拒否する。
3. While システムに他にも管理者権限を持つユーザーが存在する, the ユーザー管理機能 shall 管理者が自分自身のロール変更またはアカウント無効化を要求した場合、他ユーザーを対象とする場合と同じ挙動を適用する。

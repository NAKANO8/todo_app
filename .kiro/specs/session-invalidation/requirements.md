# Requirements Document

## Project Description (Input)
管理者（自社運用者・将来的な企業導入時のシステム管理者）は、退職者や不正利用者のアカウントを即座に締め出したいが、現状それができない。

現状: `todo-api/src/app.ts` の `@fastify/session` 登録（`app.register(session, {...})`）は `store` オプションを渡していないため、既定のインメモリ `Map` がセッションストアとして使われている。このストアは `sessionId → { authenticated, userId }` という一方向の対応表であり、リクエストのCookieから得た `sessionId` をキーに引けばそのセッションのデータ（`userId`含む）は取得できるが、逆に「`userId = 5` のセッションを全部引く」というクエリはできない——`Map` はキー（`sessionId`）でしか引けず、値の中の `userId` を逆引きするための索引をそもそも持っていないためである。

この制約により、現状ログアウトを実行できるのはリクエスト自身が自分の `session.destroy()` を呼ぶ場合のみであり、「管理者が任意の `userId` を指定して、そのユーザーの全セッションを強制的に切る」操作は、対象の `sessionId` を特定する手段がないため実行不可能である。また `Map` はNode.jsの単一プロセスのメモリ上にしか存在しないため、プロセスやコンテナを複数に増やした場合、あるインスタンスで作られたセッションは別インスタンスの `Map` からは参照できず、セッション状態がインスタンスごとに分断される。

この状態のままでは、後続で計画している管理者画面の「アカウント無効化」機能が、DB上に無効化フラグを立てるだけの見せかけの機能になってしまう。無効化されたはずのユーザーが、既存のログイン済みセッションを使って引き続きログイン状態を維持できてしまうため、管理者による強制ログアウトの手段がない。

変更内容: セッションストアを外部ストア（例: Redis）に切り替え、`userId → sessionId(群)` の逆引き索引を持たせた上で、管理者がユーザー単位でセッションを強制的に無効化できるAPI（例: `DELETE /admin/sessions/:userId`）を追加する。これにより、複数プロセス/インスタンス構成でも一貫したセッション管理と、管理者画面のアカウント無効化機能が実際に機能する基盤を提供する。

## Introduction
現在、退職者や不正利用者のアカウントを管理者が把握しても、そのユーザーが既にログイン済みのセッションを保持していれば、アカウントを無効化したところでログイン状態を強制的に終了させる手段がない。加えて、サーバーを複数のインスタンスに分けて運用した場合、セッションの状態がインスタンスごとに分断されてしまう。本機能は、管理者が特定ユーザーに紐づくセッションを強制的に無効化できる機能を提供し、後続で計画している管理者画面のアカウント無効化機能が実際に機能するための基盤を整える。

## Boundary Context (Optional)
- **In scope**: 管理者による特定ユーザー単位のセッション強制無効化、無効化操作を管理者のみに限定すること、無効化されたセッションが以降のリクエストで未認証として扱われること、既存の自己ログアウト操作が引き続き変わらず機能すること、複数サーバーインスタンス構成でも無効化結果が一貫して反映されること
- **Out of scope**: 管理者画面（ユーザー一覧・アカウント無効化UI）そのものの実装、ロールに基づく認可（グループ/チーム単位のアクセス制御）の実装、アカウント無効化ワークフロー（いつ・どのタイミングでセッション無効化を呼び出すか）の決定、無効化された本人への通知、無効化操作の監査ログ記録
- **Adjacent expectations**: 本機能は、既に導入済みの`role`（`admin`/`member`）による区別を用いて「管理者」を判定する。今後実装される管理者画面は、アカウント無効化フローの一部として本機能が提供する無効化能力を呼び出すことを期待するが、その呼び出しタイミングや画面設計は管理者画面側の責務とする。また`todo-web`側の認証結果キャッシュ（TTLを持つ）が、無効化の反映までにわずかな遅延を生む要因となるため、その遅延は許容範囲内に収まるよう調整されることを期待する。

## Requirements

### Requirement 1: 管理者によるユーザー単位のセッション強制無効化
**Objective:** As a 管理者, I want 特定ユーザーに紐づく有効なセッションをすべて無効化したい, so that 退職者や不正利用者のアカウントを即座に締め出せる

#### Acceptance Criteria
1. When 管理者が特定ユーザーのセッション無効化を要求する, the セッション管理機能 shall そのユーザーに紐づく有効なセッションをすべて無効化する。
2. When 無効化されたセッションを使ったリクエストが行われる, the セッション管理機能 shall 短い許容遅延（数秒程度）の範囲内でそのリクエストを未認証として扱う。
3. If 指定されたユーザーに有効なセッションが存在しない, then the セッション管理機能 shall エラーとせず、無効化対象が存在しなかったことを示す結果を返す。

### Requirement 2: 自己ログアウトとの独立性
**Objective:** As an 認証済みユーザー, I want 自分自身のログアウト操作が従来通り機能し続けることを期待したい, so that 本機能の導入によって既存のログアウト体験が変わらない

#### Acceptance Criteria
1. The セッション管理機能 shall 認証済みユーザー自身によるログアウト操作を、本機能導入前と同じ挙動で提供し続ける。
2. When ユーザーが自分自身の1つのセッションからログアウトする, the セッション管理機能 shall 同じユーザーが持つ他のセッション（別デバイス等）には影響を与えない。

### Requirement 3: 無効化の実行権限
**Objective:** As a システム運用者, I want 他ユーザーのセッションを強制無効化できるのは管理者のみに限定したい, so that 一般ユーザーがこの機能を悪用できないようにする

#### Acceptance Criteria
1. If 管理者権限を持たないユーザーが他ユーザーのセッション無効化を要求する, then the セッション管理機能 shall その要求を拒否し、いかなるセッションも無効化しない。
2. The セッション管理機能 shall 管理者権限を持つユーザーからの要求のみ、任意のユーザーを対象としたセッション無効化を許可する。
3. When 管理者が自分自身のセッション無効化を要求する, the セッション管理機能 shall 他ユーザーを対象とする場合と同じ挙動を適用する。

### Requirement 4: 複数サーバーインスタンス構成での一貫した無効化
**Objective:** As a システム運用者, I want サーバーを複数インスタンスに分けて運用しても無効化操作が一貫して機能することを期待したい, so that スケールした構成で無効化がインスタンスによって効いたり効かなかったりすることがない

#### Acceptance Criteria
1. While システムが複数のサーバーインスタンスで稼働している, the セッション管理機能 shall あるインスタンスで作成されたセッションを、他のすべてのインスタンスからも有効なセッションとして認識する。
2. While システムが複数のサーバーインスタンスで稼働している, the セッション管理機能 shall どのインスタンスが無効化要求を処理した場合であっても、同じ無効化結果をすべてのインスタンスに反映する。
3. The セッション管理機能 shall いずれのサーバーインスタンスに対しても、再起動を必要とせずに無効化結果を即時反映する。

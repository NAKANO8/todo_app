# Implementation Plan

- [x] 1. Foundation: usersテーブルへのroleスキーマ追加
  - `users`テーブルの定義に`role`（許容値は`admin`と`member`のみ、既定値`member`、NOT NULL）を追加する
  - 将来新しいロール値（`group_leader`等）を追加する際、既存データを書き換えずに値集合を拡張できる形にする
  - 稼働中の本番データベースには自動反映されないため、同等の変更を本番へ一度だけ手動適用する必要がある旨を記録する（デプロイ手順として扱い、本タスク一覧には含めない）
  - 開発環境のDBを作り直した際、`users`テーブルに`role`列が存在し、既定値が`member`になっていることを確認できる
  - _Requirements: 1.1, 1.2, 1.3, 2.1, 3.1_

- [x] 2. Core: アカウントデータ層でのrole取得
- [x] 2.1 ユーザーデータ取得にroleを含める
  - ロール値の許容集合を型として定義する
  - ユーザーアカウントを表す型にロール情報を含める
  - 自分自身のアカウント情報取得の際に使われるクエリにロールを含める
  - 既存アカウント（ユーザーIDでの検索）を取得すると、そのアカウントのロール値が含まれている
  - _Requirements: 1.1, 4.1_
  - _Depends: 1_

- [x] 3. Core: 自身のロール情報のAPI応答への反映
- [x] 3.1 認証済みアカウント情報の応答にroleを含める
  - 認証済みユーザーが自分のアカウント情報を取得するAPIの応答にロール値を含める
  - 認証済みユーザーとして自分のアカウント情報を取得すると、応答内容にロール値が含まれている
  - _Requirements: 4.1_
  - _Depends: 2.1_

- [x] 4. Validation: 新規登録・自己昇格防止の検証
- [x] 4.1 新規登録時のデフォルトロールと自己申告ロールの拒否を検証する
  - 新規登録後にログインし、自分のアカウント情報を取得すると、ロールが`member`であることを確認する
  - 登録時のリクエストにロール値を含めて送信しても、そのロール値が反映されないことを確認する（自己申告による昇格が成立しないことの検証）（実装時に判明: 未定義フィールドは400で拒否されず、黙って無視されて登録は201で成功する。詳細はresearch.md参照）
  - 上記2つの検証が自動テストとして実行され、いずれも合格する
  - _Requirements: 2.1, 2.2, 4.1_
  - _Depends: 3.1_

- [ ] 4.2 導入前から存在するアカウントの継続利用を検証する
  - ロール値を指定せずに作成されたアカウント（本機能導入前の既存アカウントを模したデータ）が、既定のロールを持つことを確認する
  - そのアカウントでログインおよび自分のアカウント情報取得が、本機能導入前と変わらず成功することを確認する
  - 上記の検証が自動テストとして実行され、合格する
  - _Requirements: 3.1, 3.2_
  - _Depends: 3.1_

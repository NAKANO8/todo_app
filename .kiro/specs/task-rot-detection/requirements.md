# Requirements Document

## Project Description (Input)
チームでTodoを運用するグループメンバーとリーダーの課題：タスクが誰にも気づかれないまま止まってしまうことがある。従来の「サボり検知」的な発想はメンバーに監視されている不快感を与え心理的安全性を損ない、`updated_at`のような本人が任意に書き換えられる値を基準にすると実質的な進捗がなくても信号を誤魔化せてしまう（GitHub issue #59）。

現状、Todoには期限（`due_date`）・担当者の概念がなく、`status`は本specの前提となる`task-status-model`で`pending`/`in_progress`/`blocked`/`done`に拡張される予定。グループの概念自体も`team-management`で提供される予定。

変えるべきこと：Todoに`due_date`を設定できるようにし、締切までの残り時間比率（軸A）とタスク作成〜最初のアクションまでの絶対時間（軸B）という、本人が触っても変えられない機械的な二軸で「動きが止まっている」タスクを検知する。健全→注意→警告→超過→腐敗の複数段階の色分けで視覚化し、中度・重度の腐敗はアプリ内通知（自作、消しても再出現・集約通知）で伝える。可視性はタスクの由来（アサインされたグループタスクは即公開／個人発案タスクはデフォルト非公開でヘルプ発火時に公開）で機械的に分岐させ、ユーザーが任意に切り替えられる設定にはしない。ワンタップのヘルプ機能で通知ループを止められるようにし、ヘルプ放置時は同じ腐敗ロジックを`group_leader`に再帰適用する（1段階のみ）。`blocked`は理由なしなら腐敗を継続させ、着手・却下・計画などの意思決定アクションのみ腐敗をリセットする。見積もり入力は`in_progress`移行時に求め、締切とのギャップが大きい場合は`group_leader`承認制とする。狙いは「サボり検知」ではなく「早めに動けば気持ちよく、放置すると焦る」というループを作り、放置ではなくヘルプを求める行動へ転換すること。

## Introduction
チームでタスクを運用すると、誰にも気づかれないまま停滞するタスクが生まれる。本機能は、締切（以下「締切」）からの経過時間という本人が操作できない機械的な軸でタスクの停滞を検知し、段階的な色とアプリ内通知でチームの目に自然に触れさせる。狙いは「サボりの検出」ではなく、早めに着手すれば気持ちよく、放置すると焦るという行動ループを作り、放置ではなくヘルプを求める行動へ転換することにある。

## Boundary Context (Optional)
- **In scope**: 締切の設定、締切比率（軸A）と着手ラグ（軸B）の二軸に基づく段階的な腐敗判定と色表示、`blocked`の理由付き緩和、意思決定アクションによる腐敗リセット、タスク分割による着手ラグのリセットと分割導線の表示、アプリ内通知の3段階表示と集約通知、ヘルプ機能とヘルプ放置時の`group_leader`への腐敗連鎖（1段階のみ）、タスクの由来による可視性の分岐（アサインタスクは即公開、個人発案タスクはヘルプ発火時公開）、`in_progress`移行時の見積もり入力と締切ギャップの`group_leader`承認制
- **Out of scope**: 個人発案タスクの公開・非公開をユーザーが任意に切り替えられる設定機能、組織階層をたどった多段階エスカレーション（`group_leader`のさらに上位者への通知）、見積もり精度の実績に基づく自動補正、腐敗しきい値をユーザーがカスタマイズできるUI
- **Adjacent expectations**: 本機能は`team-management`が提供するグループおよび`group_leader`ロールの存在を前提とする。本機能は`task-status-model`が確定させる`blocked`/`in_progress`等のステータス値を前提とする。本機能は`task-assignment`が確定させる担当者情報を前提とする。将来のグループチャット構想との連携（コメントによる腐敗通知やヘルプの文脈共有）は本specの範囲外だが、そのような連携が後続で行われることを妨げない設計を期待する。

## Requirements

### Requirement 1: 締切の設定
**Objective:** As a タスクの作成者, I want タスクに締切を設定したい, so that 腐敗検知の起点を持てる

#### Acceptance Criteria
1. The Task Rot Detection機能 shall タスクの作成時または編集時に締切を設定できるようにする。
2. The Task Rot Detection機能 shall 締切を設定しないままタスクを作成・編集することを許可する。
3. While タスクに締切が設定されていない, the 腐敗判定サービス shall そのタスクを締切比率に基づく判定（軸A）の対象外として扱う。

### Requirement 2: 締切比率に基づく腐敗判定と色表示（軸A）
**Objective:** As a グループメンバー, I want 締切までの残り時間の割合に応じてタスクの色が変化するのを見たい, so that 締切が近づいていることを直感的に把握できる

#### Acceptance Criteria
1. While タスクに締切が設定されており締切までの残り時間の割合が定義された軽度のしきい値以上である, the 腐敗判定サービス shall そのタスクを軽度な段階として表示する。
2. While 締切までの残り時間の割合が軽度のしきい値を下回り、かつ締切を超過していない, the 腐敗判定サービス shall そのタスクを中度な段階として表示する。
3. If タスクの締切を超過している, then the 腐敗判定サービス shall そのタスクを重度な段階として表示する。
4. When ユーザーが締切当日に初めてそのタスクへ着手する, the 腐敗判定サービス shall そのタスクの表示を中度な段階から開始する。
5. When ユーザーが締切超過後に初めてそのタスクへ着手する, the 腐敗判定サービス shall そのタスクの表示を重度な段階から開始する。

### Requirement 3: 着手ラグに基づく腐敗判定（軸B）
**Objective:** As a グループメンバー, I want 締切が遠くても作成後に放置されているタスクの停滞に気づきたい, so that 先延ばしを早期に把握できる

#### Acceptance Criteria
1. While タスクが作成されてから最初の意思決定アクションが行われていない状態が続いている, the 腐敗判定サービス shall 締切までの残り時間に関わらず、経過時間に応じて段階的に腐敗の段階を進行させる。
2. The 腐敗判定サービス shall 同一タスクについて、締切比率に基づく段階（軸A）と着手ラグに基づく段階（軸B）のうち、より腐敗が進んでいる方を表示色として採用する。
3. When 着手ラグに基づく腐敗が定義された段階に達する, the Task Rot Detection機能 shall そのタスクをより小さな単位に分割するための導線を表示する。
4. When 大きなタスクが分割され子タスクが作成される, the 腐敗判定サービス shall 各子タスクの着手ラグの起点を、その子タスクの作成時点にリセットする。

### Requirement 4: blockedタスクの腐敗継続と理由付き緩和
**Objective:** As a グループメンバー, I want blocked状態でも理由なく腐敗の進行が止まらないようにしたい, so that blockedが思考停止の隠れ蓑にならない

#### Acceptance Criteria
1. If タスクが理由を伴わずblocked状態に設定されている, then the 腐敗判定サービス shall 腐敗の進行を継続する。
2. When タスクをblocked状態に設定する際に理由と次回見直し予定が入力される, the 腐敗判定サービス shall そのタスクの腐敗の進行速度を緩和する。
3. While blocked状態の次回見直し予定を経過している, the 腐敗判定サービス shall 進行速度の緩和を終了し、通常の腐敗進行に戻す。

### Requirement 5: 意思決定アクションによる腐敗リセット
**Objective:** As a グループメンバー, I want 実質的な意思決定を行ったときだけ腐敗状態がリセットされてほしい, so that 触るだけでリセットされる誤魔化しを防げる

#### Acceptance Criteria
1. When ユーザーがタスクに対して着手・却下・計画のいずれかの意思決定アクションを行う, the 腐敗判定サービス shall そのタスクの腐敗の段階をリセットする。
2. If ユーザーがタスクに雑談的なコメントのみを追加する, then the 腐敗判定サービス shall そのタスクの腐敗の段階をリセットしない。

### Requirement 6: アプリ内通知
**Objective:** As a グループメンバー, I want 腐敗が進んだタスクについてアプリ内で通知を受け取りたい, so that 見落とさず気づける

#### Acceptance Criteria
1. While タスクの腐敗の段階が軽度である, the 通知サービス shall 個別の通知を送らず、色の変化のみで示す。
2. When タスクの腐敗の段階が中度に達する, the 通知サービス shall 画面内に通知を表示する。
3. While タスクの腐敗の段階が中度のまま放置時間が伸びている, the 通知サービス shall 通知の再表示間隔を放置時間の増加に応じて短縮する。
4. When タスクの腐敗の段階が重度に達する, the 通知サービス shall ユーザーが通知を閉じた後も、その通知を再度表示し続ける。
5. When 複数のタスクが同時に重度の腐敗に達する, the 通知サービス shall 個別の通知の代わりに、1件の集約された通知として表示する。
6. The 通知サービス shall 繰り返し表示される通知を、ユーザーがヘルプを発火するか対象のタスクを実際に進める以外の方法では止められないようにする。

### Requirement 7: ヘルプ機能
**Objective:** As a グループメンバー, I want ワンタップで助けを求めたい, so that 放置せず低摩擦でヘルプを出せる

#### Acceptance Criteria
1. The ヘルプ機能 shall ユーザーが理由の入力なしにワンタップでヘルプを発火できるようにする。
2. When ユーザーがタスクに対してヘルプを発火する, the ヘルプ機能 shall そのグループのgroup_leaderに確実に通知を届ける。
3. When ユーザーがタスクに対してヘルプを発火する, the ヘルプ機能 shall group_leader以外の一般メンバーに対してもそのタスクのヘルプ状態を閲覧可能にする。ただし一般メンバーへのメンションは行わない。
4. When ヘルプが発火される, the Task Rot Detection機能 shall そのタスクの表示を、腐敗の段階を示す色とは独立した対応中の表示に切り替える。
5. While ヘルプ発火の原因となった問題が解決していない, the Task Rot Detection機能 shall そのタスクを健全な段階として表示しない。
6. When ヘルプ発火の原因となった問題が解消される, the Task Rot Detection機能 shall そのタスクの表示を健全な段階に戻す。

### Requirement 8: ヘルプ放置時のgroup_leaderへの腐敗連鎖
**Objective:** As a group_leader, I want 自分がヘルプ対応を放置した場合にも気づけるようにしたい, so that ヘルプが二重に放置される事態を防げる

#### Acceptance Criteria
1. When ヘルプが発火される, the Task Rot Detection機能 shall そのヘルプへの対応を、group_leader宛の対応タスクとして扱う。
2. While group_leaderがヘルプへの対応を行わないまま時間が経過している, the 腐敗判定サービス shall そのヘルプ対応タスクに対しても同じ腐敗判定ロジックを適用する。
3. The Task Rot Detection機能 shall ヘルプ対応タスクの猶予時間を、通常タスクの着手ラグのしきい値よりも短いシステム固定値とする。
4. The Task Rot Detection機能 shall ヘルプ放置による腐敗のエスカレーション先をgroup_leaderの1段階のみとし、それ以上の上位者への伝播を行わない。

### Requirement 9: タスクの可視性
**Objective:** As a グループメンバー, I want タスクの由来に応じて適切な公開範囲が自動的に決まってほしい, so that 作成のたびに公開範囲を選ぶ手間なく、必要なときは自然にチームの目に触れる

#### Acceptance Criteria
1. When グループタスクとして担当者がアサインされたタスクが作成される, the Task Rot Detection機能 shall そのタスクを作成時点からグループの共有タスクボードに表示する。
2. When 個人発案のタスクが作成される, the Task Rot Detection機能 shall そのタスクをデフォルトでグループの共有タスクボードに表示しない。
3. When 個人発案タスクに対してヘルプが発火される, the Task Rot Detection機能 shall そのタスクをグループの共有タスクボードに表示する。
4. The Task Rot Detection機能 shall タスクの公開・非公開をユーザーが個別に切り替えられる設定として提供しない。
5. While ユーザーがいずれのグループにも所属していない, the Task Rot Detection機能 shall そのユーザーの個人発案タスクを他者に表示しない。

### Requirement 10: 見積もりと締切のギャップ検知
**Objective:** As a group_leader, I want 見積もりと締切の差が小さすぎる無謀な設定を承認制にしたい, so that 見積もりの過少申告による無理な計画を防げる

#### Acceptance Criteria
1. When タスクがin_progress状態に移行する, the Task Rot Detection機能 shall ユーザーに見積もり時間の入力を求める。
2. If 入力された見積もり時間と締切までの残り時間との差が定義された許容範囲を下回る, then the Task Rot Detection機能 shall そのタスクの見積もりをgroup_leaderの承認が必要な状態にする。
3. While 見積もりがgroup_leaderの承認待ちである, the Task Rot Detection機能 shall その見積もりを確定させない。
4. When group_leaderが見積もりを承認する, the Task Rot Detection機能 shall その見積もりを確定させる。

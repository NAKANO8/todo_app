# Implementation Plan

- [ ] 1. Foundation: usersテーブルのname対応と認証まわりの共通化
- [x] 1.1 usersテーブルにnameカラムを追加し、ユーザー情報の型・取得/作成処理をnameに対応させる
  - スキーマ変更(name列の追加)とそれを読み書きするコードは同一トランザクション的単位として扱う。片方だけが先にmainへ乗ると、既存のユーザー作成処理がNOT NULL制約違反で壊れるため分離しない
  - 新規DB作成時にnameカラムがNOT NULLで含まれるようにし(init.sql)、導入前から存在するアカウント向けの初期値バックフィル手順(emailのローカル部からの導出)をスキーマドキュメントに反映する。ローカルの開発DBにも同じ手順を適用する
  - ユーザーを表す型にnameを含め、ID指定でのユーザー取得・全ユーザー取得・ユーザー作成の各処理がnameを読み書きできるようにする。認証済みユーザーが自分のアカウント情報を取得する際のレスポンスにもnameを含める。パスワードハッシュは引き続きレスポンスに含めない
  - Observable: 開発DBを作り直すと新規ユーザーのレコードにnameカラムが存在し、認証済みユーザーが自分のアカウント情報を取得するとレスポンスにnameが含まれ、パスワードハッシュは含まれないことを確認できる
  - _Requirements: 1.1, 4.1, 4.2_

- [x] 1.2 (P) 認証済み判定を共通ガードとして切り出す
  - todos.route.tsに既にインラインで存在する「req.session.userIdがなければ401」の判定を、他のルートからも使える共通のpreHandlerとして切り出す。判定内容・レスポンスは変更しない
  - Observable: 既存のtodos関連の認証済み判定のテストが、リファクタ後も全て変更なくpassする
  - _Requirements: 9.1, 9.2_
  - _Boundary: requireAuthGuard_

- [ ] 2. Core: 表示名・パスワードのドメインロジック
- [ ] 2.1 (P) 新規登録時に表示名を必須項目にする
  - 登録時のリクエストスキーマにnameを必須項目として追加する(ログイン用スキーマとは分離する)。未入力または1〜50文字の範囲外の場合は登録を拒否する。登録処理がnameを保存する
  - Observable: nameを含めずに新規登録を試みると拒否され、正しく入力すると登録が成功しnameが保存されていることを確認できる
  - _Requirements: 3.1, 3.2, 3.3_
  - _Depends: 1.1_
  - _Boundary: AuthService, 登録エンドポイント_

- [ ] 2.2 (P) 自分自身の表示名変更とパスワード変更のドメインロジックを実装する
  - 認証済みユーザーが自分の表示名を新しい値に変更できるようにする。また、現在のパスワードと新しいパスワードを指定してパスワード変更を要求できるようにし、現在のパスワードが一致しない場合は拒否する。いずれの操作も対象ユーザーは常に処理を要求した本人のみとし、他ユーザーを対象にできる余地を持たせない(長さ・強度の入力検証はルート層の責務のためここでは扱わない)
  - Observable: 正しい現在パスワードでの変更要求は新しいパスワードでのログインを可能にし、誤った現在パスワードでの変更要求は拒否されることを確認できる
  - _Requirements: 2.1, 2.3, 5.1, 5.2, 7.1_
  - _Depends: 1.1_
  - _Boundary: ProfileService_

- [ ] 3. Integration: プロフィールエンドポイントを実装し、アプリケーションに登録する
- [ ] 3.1 表示名変更エンドポイントを実装し、アプリケーションに登録する
  - 表示名変更のリクエストを受け付けるエンドポイントを追加し、変更後の表示名が1〜50文字の範囲外の場合はリクエストの時点で拒否する。認証済み判定を適用してアプリケーションに登録する
  - Observable: 範囲外の長さの表示名で変更を要求すると拒否され、範囲内であれば成功することを確認できる
  - _Requirements: 2.1, 2.2, 7.1, 9.1, 9.2_
  - _Depends: 1.2, 2.2_

- [ ] 3.2 パスワード変更エンドポイントを実装し、アプリケーションに登録する
  - パスワード変更のリクエストを受け付けるエンドポイントを追加し、新しいパスワードが新規登録時と同じ強度要件を満たさない場合はリクエストの時点で拒否する。認証済み判定とログインと同水準の専用レート制限を適用してアプリケーションに登録する。パスワード変更が成功した場合、既存のセッション強制終了の能力を使って対象ユーザーの他の全セッションを終了させつつ、変更を行った本人のセッションは維持し、かつ将来の管理者による強制無効化の対象として引き続き正しく追跡されるようにする
  - Observable: パスワード変更に成功すると、変更を行った本人は引き続き操作でき、同じユーザーの別のログイン済みセッションは以降の操作で未認証として扱われることを確認できる
  - _Requirements: 5.3, 6.1, 6.2, 7.1, 8.1, 9.1, 9.2_
  - _Depends: 1.2, 2.2_

- [ ] 4. Integration: プロフィール画面と登録フォーム（todo-web）
- [ ] 4.1 (P) 登録フォームに表示名の入力欄を追加する
  - 新規登録フォームに表示名の入力欄を追加し、登録リクエストに含める。未入力または長さ範囲外の場合はクライアント側でも入力エラーを表示する
  - Observable: 登録フォームで表示名を入力せずに送信しようとするとエラーが表示され、正しく入力すると登録が成功し新しいアカウントに表示名が設定されていることを確認できる
  - _Requirements: 3.1, 3.2, 3.3_
  - _Depends: 2.1_
  - _Boundary: LoginForm, 登録プロキシルート_

- [ ] 4.2 (P) プロフィール情報を取得・更新するクライアント機能を追加する
  - 現在のユーザー情報を表す型にnameを追加し、それを取得する既存の機能を拡張する。表示名変更・パスワード変更をそれぞれ呼び出すクライアント機能を新設する
  - Observable: 表示名変更のクライアント機能を呼ぶと、実際にAPI経由で表示名が更新されることを確認できる
  - _Requirements: 1.1, 2.1, 5.1_
  - _Depends: 3.1, 3.2_
  - _Boundary: lib/types, lib/api/profile, lib/api/auth_

- [ ] 4.3 プロフィール画面のUIを実装し、Todo画面から遷移できるようにする
  - 既存の管理者画面と同じヘッダーパターン(ロゴ、タスク一覧へ戻るリンク)を持つ1ページに、「アカウント情報」(メールアドレス表示、表示名の編集、更新ボタン)と「パスワード変更」(現在のパスワード、新しいパスワード、確認用パスワードの3項目、変更ボタン)の2カードを縦に並べて実装する。確認用パスワードはクライアント側でのみ新しいパスワードとの一致を検証し、APIへは送信しない。成功・失敗はトースト通知で表示する。Todo画面のヘッダーに、ロールを問わず全認証済みユーザーが使える「プロフィール」への遷移リンクを既存の管理者リンクと同じボタンスタイルで追加する。管理者は「管理者画面」「プロフィール」「ログアウト」の3ボタンが並ぶため、ヘッダー右側のボタン群コンテナに折り返し(flex-wrap)を追加し、狭い画面幅でも崩れないようにする。未認証のユーザーは他の非公開画面と同様にアクセスできない(既存のmiddlewareの挙動により保護される。追加変更は不要)
  - Observable: Todo画面のヘッダーから「プロフィール」リンクをたどると本画面に遷移でき、表示名を変更すると画面に変更後の表示名が反映されることを確認できる
  - _Requirements: 1.1, 2.1, 5.1_
  - _Depends: 4.1, 4.2_

- [ ] 5. Validation: 回帰・整合性の確認
- [ ] 5.1 パスワード変更後のセッション整合性を確認する
  - パスワード変更に成功した際、変更を行った本人のセッションが、管理者による将来の強制セッション無効化の追跡対象として正しく維持され続けることを確認する検証を用意する(追跡対象への再登録が漏れると、本人はログインしたままだが強制無効化の対象から漏れてしまうため)
  - Observable: パスワード変更成功後、対象ユーザーに紐づく追跡対象のセッション一覧に、変更を行った本人の(新しい)セッションが含まれていることを確認できる
  - _Requirements: 6.1, 6.2_
  - _Depends: 3.2_

- [ ] 5.2 表示名・パスワード変更が要求者本人以外に影響しないことを確認する
  - 認証済みユーザーが表示名変更・パスワード変更を要求した際、他ユーザーのアカウントが一切変更されないことを確認する検証を用意する
  - Observable: あるユーザーとしてログインした状態で操作を行っても、別ユーザーの表示名・パスワードが変化しないことを確認できる
  - _Requirements: 7.1_
  - _Depends: 3.1, 3.2_

- [ ] 5.3 導入前から存在するアカウントへの表示名バックフィルを確認する
  - 表示名を持たない既存アカウントに対し、設計で定義したバックフィル手順(emailのローカル部からの導出)を適用した結果を確認する検証を用意する
  - Observable: バックフィル手順適用後、対象アカウントの表示名がそのアカウントのemailの@より前の部分と一致することを確認できる
  - _Requirements: 4.1, 4.2_
  - _Depends: 1.1_

- [ ] 5.4 パスワード変更エンドポイントのレート制限を確認する
  - 短時間に大量のパスワード変更試行が行われた場合に、それ以上の要求が一時的に拒否されることを確認する検証を用意する
  - Observable: 設定した回数を超えてパスワード変更を試みると、それ以降の要求が拒否されることを確認できる
  - _Requirements: 8.1_
  - _Depends: 3.2_

## Implementation Notes
- **`/kiro-impl`実行前に判明**: 元のタスク一覧では1.1(nameカラム追加のみ)と1.3(型・リポジトリのnameコード対応)が別タスクだったが、CI(`mysql/init.sql`を毎回フレッシュ適用)・ローカル開発DBのいずれでも、1.1単独がmainへ乗ると既存の`createUser`(まだnameを渡さない)がNOT NULL制約違反で失敗し、既存テストの回帰を引き起こすことが分かった。スキーマとそれを書き込むコードは分離不可能な単位のため1.1に統合し、1.3は廃止した。ローカル開発DB(Docker、永続ボリューム)は`init.sql`の変更を自動では反映しないため、1.1の実装時に手動で同じ3段階のALTER/UPDATEを適用する。

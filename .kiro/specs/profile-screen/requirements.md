# Requirements Document

## Project Description (Input)
プロフィール画面の最低限実装。ログイン中のユーザーが自分のプロフィール画面で表示名(name)を確認・変更でき、パスワードを変更できる。アバター画像アップロードは対象外(将来のUI改善フェーズで対応)。todos-app-internal/todo-app-roadmap.mdの「第一優先：OSS＋オンプレ配布方針」内タスクに基づく。

### nameフィールドの設定・変更方針(確定事項)
- 現状、登録フォーム(email/passwordのみ)にはnameを入力する手段が存在しない(コード確認済み: `todo-web/features/auth/LoginForm.tsx`)
- **登録時**: nameを必須入力項目として登録フォームに追加する。新規ユーザーは登録完了時点で必ずnameを持つ
- **既存ユーザー(本機能導入前に登録済み)**: emailのローカル部(`@`より前の部分)から自動生成した値をデフォルトのnameとしてバックフィルする
- **プロフィール画面**: nameの閲覧に加えて**変更も可能**とする(当初想定していた「閲覧のみ」から変更。ユーザーの明示的な希望により編集可能フィールドとする)

### スコープ確定の経緯・明記すべき制約

**パスワード変更はメール認証の有無に依存しない**
- 「ログイン中の本人がcurrent passwordを再確認して変更する」フローと、「メールリンクによるパスワードリセット」(ロードマップ項目9、別スペック)は別物。本人確認をメールに依存しないため、メール認証未実装でも安全に実装できる
- ただし変更成功時にセッションを乗っ取られた場合の被害を抑えるため、以下2点を必須とする:
  1. 変更には**現在のパスワードの再入力**を必須とする
  2. 変更成功時、既存の`SessionService.invalidateUserSessions`パターンを用いて**当該ユーザーの他の全セッションを強制無効化**する

**IDOR対策(必須の受け入れ基準)**
- 既存コード(`todos.controller.ts`等)は全て`req.session.userId`のみで対象ユーザーを特定しており、クライアント指定のIDを信用する箇所は管理者専用エンドポイント以外に存在しない
- パスワード変更エンドポイントも同じパターンを踏襲し、**リクエストボディ等のuserIdを一切受け付けず、セッションのみで対象ユーザーを特定する**こと

**専用レート制限(必須の受け入れ基準)**
- 現状、`/auth/login`(15分10回)・`/auth/register`(1時間5回)には専用レート制限があるが、それ以外のルートは全体共通200回/分(IP単位)のみ
- current passwordの再入力は実質的にログインと同じ総当たり対象になるため、パスワード変更エンドポイントには**ログインと同水準の専用レート制限**を設定すること

**`name`フィールドの入力検証(必須の受け入れ基準)**
- `users`テーブルに`name`カラムを新規追加する。既存の`todos.title`と同水準(`minLength:1`, 上限文字数指定, `additionalProperties:false`)のAJVスキーマによる長さ・型のバリデーションを行うこと

**明示的にスコープ外とする既存の全体課題**
- 認証・Cookie関連の既存セキュリティ設定の見直し: profile screen固有の課題ではなくアプリ全体に関わる検討事項であるため、本specのスコープには含めない。詳細は非公開の管理ドキュメントで別途扱う
- アバター画像アップロード: 将来のUI改善フェーズで対応

## Introduction
現在、ユーザーは自分の表示名を確認・変更する手段を持たず、パスワードもログイン後に自分で変更する手段がない。新規登録時にも表示名を入力する項目がなく、本機能導入前から存在するアカウントは表示名を一切持たない状態にある。本機能は、ログイン中のユーザーが自分の表示名を確認・変更し、パスワードを安全に変更できる最低限のプロフィール画面を提供する。

## Boundary Context (Optional)
- **In scope**: 自分自身の表示名の閲覧・変更、新規登録時の表示名入力の必須化、本機能導入前から存在するアカウントへの表示名デフォルト値の設定、自分自身のパスワードの変更(現在のパスワードの確認を伴う)、パスワード変更成功時の他セッションの強制終了、表示名変更・パスワード変更を要求者本人のアカウントのみに限定すること、パスワード変更操作に対する総当たり対策
- **Out of scope**: アバター画像アップロード、パスワード変更のメール通知、メールリンクによるパスワードリセット(未ログイン状態からの復旧、別スペックの対象)、他ユーザーの表示名・パスワードの閲覧や変更、認証・Cookie関連の既存セキュリティ設定の見直し(アプリ全体に関わる既存の検討事項であり本機能固有ではないため別トラックで扱う)
- **Adjacent expectations**: 本機能は、既に導入済みのセッション強制終了の能力(`session-invalidation` spec)を利用してパスワード変更時に他セッションを終了させることを期待するが、具体的な連携方法は設計フェーズで定める。パスワードの強度要件は新規登録時に適用されているものを踏襲し、本機能で新たに定義しない。

## Requirements

### Requirement 1: 自分自身の表示名の閲覧
**Objective:** As a ログイン中のユーザー, I want 自分の表示名を確認したい, so that 自分のアカウントにどの表示名が設定されているかを把握できる

#### Acceptance Criteria
1. When ログイン中のユーザーがプロフィール画面を開く, the プロフィール機能 shall 現在設定されている表示名を提示する。

### Requirement 2: 自分自身の表示名の変更
**Objective:** As a ログイン中のユーザー, I want 自分の表示名を変更したい, so that 自分の情報を最新の状態に保てる

#### Acceptance Criteria
1. When ログイン中のユーザーが新しい表示名への変更を要求する, the プロフィール機能 shall 要求者本人の表示名を指定された値に更新する。
2. If 変更後の表示名が1文字未満または50文字を超える, then the プロフィール機能 shall その変更要求を拒否する。
3. When 表示名が変更される, the プロフィール機能 shall 変更後、そのユーザーが次に自分のアカウント情報を取得した際に新しい表示名を反映する。

### Requirement 3: 新規登録時の表示名入力
**Objective:** As a 新規登録するユーザー, I want 登録時に自分の表示名を設定したい, so that 登録直後から表示名が設定された状態でサービスを利用できる

#### Acceptance Criteria
1. When ユーザーが新規登録を行う, the プロフィール機能 shall 表示名の入力を必須項目として要求する。
2. If 表示名が入力されずに新規登録が要求される, then the プロフィール機能 shall その登録要求を拒否する。
3. If 入力された表示名が1文字未満または50文字を超える, then the プロフィール機能 shall その登録要求を拒否する。

### Requirement 4: 導入前から存在するアカウントへの表示名の設定
**Objective:** As a システム運用者, I want 本機能導入前から存在するアカウントにも表示名を設定しておきたい, so that 既存ユーザーもプロフィール画面を問題なく利用できる

#### Acceptance Criteria
1. When 本機能が導入される, the プロフィール機能 shall 導入前から表示名を持たない既存アカウントそれぞれに対し、そのアカウントのメールアドレスから機械的に導出した値を初期の表示名として設定する。
2. While 導入前から存在するアカウントの表示名が本機能導入後に変更されていない, the プロフィール機能 shall 導入時に設定された初期の表示名を維持する。

### Requirement 5: 自分自身のパスワードの変更
**Objective:** As a ログイン中のユーザー, I want 自分のパスワードを変更したい, so that パスワードが漏洩した疑いがある場合や定期的な更新のために自分でパスワードを更新できる

#### Acceptance Criteria
1. When ログイン中のユーザーが新しいパスワードへの変更を要求する, the プロフィール機能 shall 要求に現在のパスワードが含まれ、かつそれが登録済みのパスワードと一致することを条件に、要求者本人のパスワードを更新する。
2. If 変更要求に含まれる現在のパスワードが登録済みのパスワードと一致しない, then the プロフィール機能 shall そのパスワード変更要求を拒否する。
3. If 新しいパスワードが新規登録時に適用されるパスワード強度要件を満たさない, then the プロフィール機能 shall そのパスワード変更要求を拒否する。

### Requirement 6: パスワード変更に伴う他セッションの強制終了
**Objective:** As a ログイン中のユーザー, I want パスワードの変更が成功したら他の場所でログインしたままのセッションを終了させたい, so that 自分の知らないセッションが乗っ取られていた場合の被害を抑えられる

#### Acceptance Criteria
1. When パスワードの変更が成功する, the プロフィール機能 shall 変更を行った本人に紐づく、変更操作を行ったセッション以外の既存のログイン済みセッションをすべて強制的に終了させる。
2. When パスワードの変更が成功する, the プロフィール機能 shall 変更操作を行った本人のセッションについては、変更後も継続してログイン状態を維持する。

### Requirement 7: プロフィール操作対象の本人限定
**Objective:** As a システム運用者, I want 表示名の変更・パスワードの変更のいずれについても要求者本人のアカウントのみを対象にしたい, so that 他ユーザーのアカウントが意図せず変更される事態を防げる

#### Acceptance Criteria
1. The プロフィール機能 shall 表示名の変更・パスワードの変更のいずれについても、要求元の認証済みユーザー本人のアカウントのみを変更対象とし、要求に含まれる他の情報によって対象が変わることがないようにする。

### Requirement 8: パスワード変更操作への総当たり対策
**Objective:** As a システム運用者, I want パスワード変更操作に対して短時間の大量試行を制限したい, so that 現在のパスワードを推測する総当たり攻撃からユーザーを保護できる

#### Acceptance Criteria
1. While 同一の要求者からパスワード変更の要求が短時間に繰り返される, the プロフィール機能 shall 既存のログイン試行に適用されている制限と同等の水準で、それ以上の要求を一時的に拒否する。

### Requirement 9: 未認証ユーザーからのアクセス制限
**Objective:** As a システム運用者, I want 未認証のユーザーがプロフィール画面の閲覧・表示名変更・パスワード変更のいずれも行えないようにしたい, so that ログインしていない第三者が他人のアカウント情報にアクセスできないようにする

#### Acceptance Criteria
1. If 未認証のユーザーがプロフィール画面の表示、表示名の変更、またはパスワードの変更のいずれかを要求する, then the プロフィール機能 shall その要求を拒否する。
2. The プロフィール機能 shall プロフィール画面の表示、表示名の変更、パスワードの変更のいずれについても、認証済みユーザーからの要求のみを許可する。

# Requirements Document

## Project Description (Input)
チームでこのアプリを使いたい管理者・グループリーダーは、現状「誰が同じグループに属するか」「グループ単位でメンバーの状況を把握する」手段を持たない。`users`テーブルには`role`（`admin`/`member`）と`status`（有効/無効）のみが存在し、グループやチームという概念自体がない。`role`カラムはenum拡張に耐える設計になっており、`group_leader`はグループ機能実装時に追加することが`admin-role` spec時点（2026-07-01）で明記されている。また`admin-user-management` specでも、グループ/チーム単位のアクセス制御は別イテレーションで扱うと明記されており、本specがそれに当たる。

グループ（チーム）という単位を導入し、ユーザーはいずれかのグループに所属する。`role`のenumに`group_leader`を追加し、`group_leader`ロールを持つユーザーは自分が率いるグループのメンバー一覧と各メンバーの状態を把握できるようにする。グループの作成・メンバーの追加/削除は`admin`権限を持つユーザーが行う。`group_leader`は自グループのメンバー一覧を参照できるが、他グループのメンバーは参照できない。

本specはグループ・メンバーシップ・可視性の権限基盤のみを提供する。実際に何を見せるか（腐敗信号など）は`task-rot-detection`specの責務であり、Todoの内容やステータスをグループリーダーに見せる機能自体は本specの対象外とする。1ユーザーが複数グループに所属するケース、グループの階層構造、グループ単位の詳細な権限カスタマイズ（read/manage分離）も対象外とする。

本機能は`orm-migration`完了後のスキーマ移行を前提として着手し、既存の`admin`/`member`の挙動・認可ルールを壊さないこと。

## Boundary Context (Optional)
- **In scope**: グループの作成・改称・削除（管理者操作）、全ユーザーが常にいずれか1つのグループに所属する状態の維持（デフォルトグループを含む）、グループメンバーシップの割り当て・除外（管理者操作）、`role`への`group_leader`追加、`group_leader`による自グループメンバー一覧の参照、グループスコープの認可（自グループ外のメンバー情報へのアクセス拒否）
- **Out of scope**: グループが所有するTodoの内容・ステータスの可視化（`task-rot-detection`の責務）、1ユーザーが複数グループに所属するケース、グループの階層構造（グループのグループ等）、グループ単位の権限カスタマイズ（read/manage分離等）、ユーザー一覧の検索・絞り込み・ページネーション（`admin-user-management`のスコープ外のまま）
- **Adjacent expectations**: 本機能は`orm-migration`によるスキーマ移行の完了後に導入される。グループからの除外（メンバーシップ変更）に伴うセッション強制ログアウトの要否は本specでは決定せず、設計フェーズで検討する。`task-assignment`・`task-rot-detection`は本機能が提供するグループ・メンバーシップ・グループスコープ認可の基盤を利用する想定だが、それらの機能自体は本specの対象外である。既存の`admin`/`member`の挙動（ユーザー一覧参照、ロール変更、アカウント状態変更、「有効な管理者が最低1人残る」不変条件）は本機能によって変更されない。

## Requirements

### Requirement 1: グループの作成・改称・削除
**Objective:** As a 管理者, I want グループの作成・改称・削除を行いたい, so that チーム単位でメンバーを編成できる

#### Acceptance Criteria
1. When 管理者がグループ名を指定してグループの作成を要求する, the グループ管理機能 shall 指定された名前を持つ新しいグループを作成する
2. If グループ名が空文字列で指定される, then the グループ管理機能 shall グループの作成を拒否し、エラーを返す
3. When 管理者が既存グループの名称変更を要求する, the グループ管理機能 shall 対象グループの名称を更新する
4. When 管理者が所属メンバーが0人のグループの削除を要求する, the グループ管理機能 shall 当該グループを削除する
5. If 管理者が1人以上のメンバーが所属するグループの削除を要求する, then the グループ管理機能 shall 削除を拒否し、エラーを返す
6. If 管理者以外のユーザーがグループの作成・改称・削除を要求する, then the グループ管理機能 shall その要求を拒否する

### Requirement 2: 全ユーザーの常時グループ所属
**Objective:** As a システム運用者, I want 全ユーザーが常にいずれか1つのグループに所属している状態を維持したい, so that グループスコープの認可が一貫して機能する

#### Acceptance Criteria
1. The グループ管理機能 shall 本機能の導入時に、既存の全ユーザーを自動的に生成されたデフォルトグループへ割り当てる
2. When 新しいユーザーアカウントが作成される, the グループ管理機能 shall そのユーザーをデフォルトグループへ自動的に割り当てる
3. The グループ管理機能 shall 全てのユーザーアカウントが常にちょうど1つのグループに所属する状態を保証する

### Requirement 3: グループメンバーシップの管理
**Objective:** As a 管理者, I want ユーザーを任意のグループへ割り当てたい, so that チーム編成の変更を反映できる

#### Acceptance Criteria
1. When 管理者がユーザーとグループを指定して割り当てを要求する, the グループ管理機能 shall 当該ユーザーの所属グループを指定されたグループに変更する
2. If 割り当て先のグループが存在しない, then the グループ管理機能 shall 割り当てを拒否し、エラーを返す
3. When 管理者がグループからのメンバー除外を要求する, the グループ管理機能 shall 当該ユーザーの所属グループをデフォルトグループに変更する
4. If 管理者以外のユーザーがメンバーシップの割り当て・除外を要求する, then the グループ管理機能 shall その要求を拒否する

### Requirement 4: group_leaderロールの追加
**Objective:** As a 管理者, I want ユーザーに`group_leader`ロールを付与したい, so that そのユーザーが自グループのメンバーを見守れるようにする

#### Acceptance Criteria
1. The グループ管理機能 shall ユーザーのロールとして`admin`・`member`・`group_leader`の3値を扱う
2. When 管理者がユーザーのロールを`group_leader`へ変更することを要求する, the グループ管理機能 shall 当該ユーザーのロールを更新する
3. If 管理者以外のユーザーがロール変更を要求する, then the グループ管理機能 shall その要求を拒否する
4. The グループ管理機能 shall `group_leader`ロールを持つユーザーについて、そのユーザーが現在所属しているグループを「自グループ」として扱う
5. When `group_leader`ロールを持つユーザーの所属グループが変更される, the グループ管理機能 shall そのユーザーが参照できる対象を新しい所属グループに更新する

### Requirement 5: group_leaderによる自グループメンバー一覧の参照
**Objective:** As a group_leader, I want 自グループのメンバー一覧と各メンバーの状態を確認したい, so that チームの状況を把握できる

#### Acceptance Criteria
1. When `group_leader`ロールを持つユーザーが自グループのメンバー一覧を要求する, the グループ管理機能 shall 自グループに所属する全メンバーの一覧（氏名・メールアドレス・アカウント状態）を返す
2. The グループ管理機能 shall `group_leader`自身を一覧に含める
3. While 自グループに所属するメンバーが`group_leader`自身のみである, the グループ管理機能 shall `group_leader`自身のみを含む一覧を返す

### Requirement 6: グループスコープの認可
**Objective:** As a 製品オーナー, I want 自グループ以外のメンバー情報にアクセスできないようにしたい, so that チーム間でメンバー情報が漏えいしない

#### Acceptance Criteria
1. If `group_leader`ロールを持つユーザーが自グループ以外のグループのメンバー一覧を要求する, then the グループ管理機能 shall その要求を拒否する
2. If `member`ロールを持つユーザーがグループのメンバー一覧の参照を要求する, then the グループ管理機能 shall その要求を拒否する
3. If 未認証のユーザーがグループのメンバー一覧の参照を要求する, then the グループ管理機能 shall その要求を拒否する
4. The グループ管理機能 shall `admin`ロールを持つユーザーに対し、いずれのグループのメンバー一覧も参照できることを許可する

### Requirement 7: 既存admin/member機能の非回帰
**Objective:** As an 既存の管理者, I want `group_leader`ロール追加後も既存の`admin`/`member`機能が変わらず動作することを確認したい, so that 既存の運用が壊れない

#### Acceptance Criteria
1. The グループ管理機能 shall 既存の管理者専用機能（ユーザー一覧参照、ロール変更、アカウント状態変更）の外部から見た挙動を変更しない
2. The グループ管理機能 shall 「有効な管理者が最低1人残る」という既存の不変条件を、`group_leader`ロールの追加によって変化させない
3. If ロールを`admin`から`group_leader`または`member`へ変更する要求によって有効な管理者が0人になる, then the グループ管理機能 shall その要求を拒否する

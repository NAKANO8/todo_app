# Implementation Plan

- [ ] 1. Foundation: グループスキーマとマイグレーション基盤
- [ ] 1.1 groupsテーブルとデフォルトグループの導入
  - `schema.prisma`に`Group`モデル（`id`, `name`, `isDefault`, `createdAt`, `updatedAt`）を定義する
  - マイグレーションで`groups`テーブルを作成し、`isDefault = true`のデフォルトグループを1行seedする
  - 観測可能な完了状態: 新規の空DBに対してマイグレーションを適用すると、`groups`テーブルに`isDefault = true`の行が1つだけ存在する
  - _Requirements: 2.1_

- [ ] 1.2 既存ユーザーおよび新規ユーザーの常時グループ所属の確立
  - `users.group_id`列をNULL許容で追加した上で、既存の全ユーザー行にデフォルトグループのidをbackfillし、その後`NOT NULL`化・`groups`への外部キー制約を追加する
  - `users.group_id`にデフォルトグループの予約idをDBレベルの既定値として設定する
  - 観測可能な完了状態: マイグレーション適用後、既存の全ユーザー行の`group_id`がデフォルトグループを指しており、`group_id`を指定しない新規ユーザー作成でも自動的にデフォルトグループが設定される
  - _Requirements: 2.1, 2.2, 2.3_
  - _Depends: 1.1_

- [ ] 1.3 group_leaderロールの追加
  - Prismaの`UserRole` enumに`group_leader`を追加し、`todo-api`内のTypeScript側`UserRole`型も同時に更新する
  - 観測可能な完了状態: `prisma generate`が成功し、`admin`/`member`/`group_leader`の3値を受け付ける`UserRole`型が生成される
  - _Requirements: 4.1_
  - _Depends: 1.1_

- [ ] 1.4 マイグレーション適用の検証
  - 新規の空DBに対してマイグレーションを適用し、期待するテーブル構造・デフォルトグループ・DBレベル既定値が得られることを確認する
  - 移行前の`users`データを模したフィクスチャ（`group_id`列が存在しない状態）にマイグレーションを適用し、全既存ユーザーの`group_id`がデフォルトグループを指すことを確認する
  - 観測可能な完了状態: 上記2パターンのマイグレーション適用検証がともに成功する
  - _Requirements: 2.1, 2.2_
  - _Depends: 1.1, 1.2_

- [ ] 2. (P) グループ管理ドメインの追加（管理者向けCRUD）
  - `groups`テーブルという新しいデータ主体に対応する新規ドメイン。他の並行タスク（3, 4）とはファイルが重複しないため並行実施可能。ただし2.1・2.2は同一ファイル群を編集するため内部は順次実施する
- [ ] 2.1 グループの作成・一覧取得
  - グループの作成（グループ名を受け取り新規グループを作成する。空文字列の場合は作成を拒否する）と、全グループの一覧取得を実装する
  - 管理者以外からのアクセスを拒否する（既存の管理者専用ガードを再利用する）
  - 観測可能な完了状態: 有効な名前でグループ作成を要求すると新しいグループが作成され、空文字列を指定すると拒否される。管理者以外が要求すると拒否される
  - _Requirements: 1.1, 1.2, 1.6_
  - _Boundary: GroupRepository, AdminGroupService, AdminGroupController_

- [ ] 2.2 グループの改称・削除
  - 既存グループの名称変更を実装する（存在しない場合はエラー）
  - グループの削除を実装し、デフォルトグループの削除、および1人以上のメンバーが所属するグループの削除をいずれも拒否する
  - 観測可能な完了状態: メンバーが0人の非デフォルトグループのみ削除に成功し、デフォルトグループまたは1人以上のメンバーがいるグループの削除要求はエラーになる
  - _Requirements: 1.3, 1.4, 1.5, 2.3_
  - _Boundary: GroupRepository, AdminGroupService, AdminGroupController_
  - _Depends: 2.1_

- [ ] 3. (P) ユーザーのグループ所属情報へのデータアクセス追加
  - 既存の`users`テーブル担当リポジトリに、グループ関連の読み書き（対象ユーザーの所属グループ変更、指定グループに所属する全ユーザーの取得）を追加する。既存の認証・ロール変更・ステータス変更・プロフィール関連メソッドの挙動は一切変更しない
  - 観測可能な完了状態: 指定したユーザーの所属グループを変更する処理と、指定したグループに所属する全ユーザー（氏名・メールアドレス・アカウント状態を含み、パスワードハッシュを含まない）を取得する処理が、それぞれ独立したテストで確認できる
  - _Requirements: 3.1, 3.3, 4.1, 4.4, 5.1, 5.2, 6.4, 7.1, 7.2, 7.3_
  - _Boundary: AuthRepository_
  - _Depends: 1.2, 1.3_

- [ ] 4. (P) group_leader認可ガードの追加
  - 認証済みであり、かつロールが`group_leader`であることを確認する共有の認可チェックを追加する（既存の管理者専用ガードと同型の判定パターンに従う）
  - 未認証は拒否、`member`/`admin`ロールも拒否する
  - 観測可能な完了状態: `group_leader`ロールを持つユーザーのみがこのガードを通過し、それ以外のロールおよび未認証リクエストは拒否される
  - _Requirements: 6.2, 6.3_
  - _Boundary: GroupLeaderOnlyGuard_
  - _Depends: 1.3_

- [ ] 5. (P) 管理者によるグループ再割り当てとgroup_leaderロール付与のAPI拡張
  - 既存の管理者向けユーザー操作（ユーザー一覧・ロール変更・ステータス変更）のエンドポイント群に、対象ユーザーの所属グループを変更する操作を追加する。割り当て先グループが存在しない場合はエラーとし、管理者以外からの要求は拒否する
  - 既存のロール変更エンドポイントが受け付ける値に`group_leader`を追加する
  - グループの再割り当ておよびロール変更（`group_leader`への変更を含む）は、セッションの強制無効化を伴わない（既存の`admin`/`member`間のロール変更と同じ挙動）
  - 観測可能な完了状態: 存在するグループへの再割り当てが成功し、存在しないグループを指定するとエラーになる。ロール変更要求で`group_leader`を指定すると成功する
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 4.2, 4.3_
  - _Boundary: AdminUserService, AdminUserController, admin.user.route.ts_
  - _Depends: 2.1, 3_

- [ ] 6. (P) group_leaderによる自グループメンバー一覧API
  - `group_leader`ロールを持つユーザーが、自分自身が現在所属しているグループのメンバー一覧（自分自身を含む、氏名・メールアドレス・アカウント状態）を取得できるエンドポイントを追加する
  - このエンドポイントは要求者自身の所属グループのみを対象とし、他グループを指定してアクセスする経路を一切持たない
  - 所属グループが変更された場合、次回以降の一覧取得は新しい所属グループのメンバーを返す
  - 観測可能な完了状態: `group_leader`が一覧を要求すると自グループの全メンバー（自身のみの場合も含む）が返り、他グループの情報は一切含まれない
  - _Requirements: 4.4, 4.5, 5.1, 5.2, 5.3, 6.1_
  - _Boundary: GroupLeaderService, GroupLeaderController_
  - _Depends: 3, 4_

- [ ] 7. (P) 管理者による任意グループのメンバー一覧参照API
  - 既存の管理者向けグループ管理エンドポイント群に、指定したグループのメンバー一覧を取得する操作を追加する
  - 観測可能な完了状態: 管理者はいずれのグループを指定してもそのメンバー一覧を取得できる。存在しないグループを指定するとエラーになる
  - _Requirements: 6.4_
  - _Boundary: AdminGroupService, AdminGroupController_
  - _Depends: 2.1, 3_

- [ ] 8. Validation: 非回帰・グループスコープ認可・ライフサイクルの統合確認
- [ ] 8.1 既存admin/member機能の非回帰確認
  - 既存のユーザー一覧参照・ロール変更・ステータス変更に関するテストが、`group_id`列・`group_leader`ロール追加後も変更なしに合格することを確認する
  - ロールを`admin`から`group_leader`または`member`に変更する要求によって有効な管理者が0人になる場合、要求が拒否されることを確認するテストを追加する
  - 観測可能な完了状態: 既存の管理者関連テストスイート全体が合格し、「有効な管理者が最低1人残る」不変条件が`group_leader`追加後も維持されることが新規テストで確認できる
  - _Requirements: 7.1, 7.2, 7.3_
  - _Depends: 5_

- [ ] 8.2 グループスコープ認可の統合テスト
  - `group_leader`が自グループ以外のメンバー情報に一切アクセスできないこと、`member`ロールおよび未認証ユーザーがメンバー一覧参照エンドポイントを拒否されること、`admin`はいずれのグループのメンバー一覧も参照できることを、実際のガード・エンドポイントに対して確認する
  - 観測可能な完了状態: 上記の認可条件（許可・拒否の組み合わせ）がすべてテストで確認できる
  - _Requirements: 6.1, 6.2, 6.3, 6.4_
  - _Depends: 6, 7_

- [ ] 8.3 グループライフサイクルの統合テスト
  - グループ作成 → メンバーの割り当て → `group_leader`としての自グループメンバー一覧参照 → デフォルトグループへの除外（再割り当て）→ 空になったグループの削除、という一連の流れを実際のエンドポイントに対して確認する
  - 観測可能な完了状態: 一連の操作を順に実行した結果が各段階で期待通りになり、最終的にグループが削除できることを確認できる
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 2.1, 2.2, 2.3, 3.1, 3.2, 3.3, 3.4, 4.4, 4.5, 5.1, 5.2, 5.3_
  - _Depends: 5, 6, 8.1_
